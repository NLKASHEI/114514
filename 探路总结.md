# 道渊配置小助手 — 探路总结

## 1. iframe 里的 SillyTavern 是什么

JS-Slash-Runner 在 `src/iframe/predefine.js` 中定义了 iframe 的 `window.SillyTavern`：

```javascript
Object.defineProperty(window, 'SillyTavern', {
  get: () => {
    const SillyTavern = _.get(window.parent, 'SillyTavern');
    const getContext = () => {
      return { ...SillyTavern.getContext(), writeExtensionField: _th_impl.writeExtensionField };
    };
    return { ...getContext(), getContext };
  },
});
```

每次访问 `SillyTavern` 都会触发 getter，调用 `parent.SillyTavern.getContext()` 获取上下文快照。

**关键点**：`getContext()` 返回的对象中，`extensionSettings` 是**引用**而非拷贝。所以 `SillyTavern.extensionSettings` 始终指向父页面的实时对象。

## 2. 为什么不能用 `parent.SillyTavern` 直接读

`parent.SillyTavern` 只有两个键：`libs` 和 `getContext`。没有 `extensionSettings`、没有 `getChatCompletionModel`。所有业务属性都必须通过 `getContext()` 获取。

因此 iframe 中正确做法是：**直接用 `SillyTavern`（走 getter），不要用 `parent.SillyTavern`**。

## 3. 为什么不能用 `runInParent` 做简单读取

`runInParent` 是往父页面注入 `<script>` 标签执行代码，适用于写操作（如 `TavernHelper.replaceWorldbook`）。

对于读操作（如读 `extensionSettings`），直接访问 iframe 的 `SillyTavern` 即可获得实时引用，无需注入脚本。注入脚本反而有额外开销和上下文切换。

## 4. 主插头 API URL 的位置

| 路径 | 值 | 说明 |
|------|-----|------|
| `extensionSettings.apiUrl` | `http://localhost:5100` | 默认值，不是实际连接地址 |
| `extensionSettings.connectionManager.selectedProfile` | profile ID | 当前选中的连接配置 ID |
| `extensionSettings.connectionManager.profiles[n]['api-url']` | 真实 API URL | 选中 profile 的 `api-url` 字段才是主插头的真实地址 |

**正确读法**：
```javascript
function getMainApiUrl() {
  const cm = SillyTavern.extensionSettings.connectionManager;
  const prof = cm.profiles.find(p => p.id === cm.selectedProfile);
  return prof ? prof['api-url'] : '';
}
```

## 5. DES 加密码的正确构建方式

```javascript
const model = SillyTavern.getChatCompletionModel();  // 主插头模型名
const apiUrl = getMainApiUrl();  // 主插头 API URL
const payload = apiUrl ? (model + '|' + apiUrl) : model;
// 然后用 DES 加密 payload
```

**之前的错误**：读的是 `mvu_settings.额外模型解析配置.api地址`（MVU 额外模型的 API），跟主插头不是同一个东西。

## 6. JS-Slash-Runner 注入的所有全局对象

从 `predefine.js` 可以看出，以下对象直接挂载到了 iframe 的 `window`：

### 6.2 TavernHelper._bind 绑定

```javascript
Object.entries(TavernHelper._bind).map(([key, value]) => ({
  [key.replace('_', '')]: value.bind(window),
}));
```

`TavernHelper` 的 `_bind` 里的函数会被去掉 `_` 前缀后绑定到 iframe 的 `window`。例如 `_eventOn` → `eventOn`。

### 6.3 Mvu

```javascript
Object.defineProperty(window, 'Mvu', {
  get: () => _.get(window.parent, 'Mvu'),
});
```

`Mvu` 是 getter，直接读取 parent 上的 `Mvu` 对象（不是走 `getContext`）。

### 6.4 CryptoJS

`CryptoJS` **不在** iframe 的 `window` 上。用 `parent.CryptoJS` 或 `window.parent.CryptoJS` 获取。

## 11. 旧版酒馆兼容层

探路验证脚本在旧版 ST 上跑出 41/48 通过（新版 51/51），以下7项为核心差异。

### 11.1 新旧版本差异对照

| 项目 | 新版 ST (57键) | 旧版 ST (37键) |
|------|---------------|---------------|
| extensionSettings 键数 | 57 | 37 |

| connectionManager.profiles | 有 profile | 0 个 profile |
| EjsTemplate 扩展 | ✅ 存在 | ❌ 不存在 |

### 11.2 DES 加密：纯 JS 回退

旧版没有 `parent.CryptoJS`，主文件内置了完整 DES-ECB-PKCS7 纯 JS 实现（约90行），`encryptPayload()` 自动选择 CryptoJS 优先、纯 JS 回退。

### 11.3 API URL：三级回退

```javascript
function getMainApiUrl() {
  // 1. connectionManager 选中 profile（新版主路径）
  // 2. chatCompletionSettings 中 URL 键（旧版回退）
  // 3. 遍历 chatCompletionSettings 所有值，找 http 开头（兜底）
```

### 11.4 模型名回退

`checkConfig()` 中 `getChatCompletionModel()` 返回空时，调用 `inferModelFromSettings()` 从 `chatCompletionSettings` 的 source 映射键（如 `deepseek_model`、`google_model`）推断模型名。

### 11.5 saveSettingsDebounced 多路径

```javascript
function saveSettings() {
  const fn = SillyTavern.saveSettingsDebounced
    || (p.SillyTavern && p.SillyTavern.saveSettingsDebounced)
    || (typeof p.saveSettingsDebounced === 'function' ? p.saveSettingsDebounced : null);
  if (fn) return fn();
  throw new Error('saveSettingsDebounced 不可用');
}
```

旧版该函数可能挂在不同位置，多路径兜底。

### 11.6 移动端悬浮窗

移动端视口 360x845 测试结果：`z-index: 1000000` 足够穿透所有图层，两个探针圆 (z=9999999/1000000) 均可见。移动端问题通常不是图层，而是脚本未加载（如 git tag 缺失导致 jsDelivr 404）。

## 13. 手动配置保存机制探路（2026-05-25）

### 13.1 问题现象

手动配置面板中修改任意字段 → 点"应用" → 页面刷新后**全部回到 MVU 框架出厂默认值**。但"一键最优配置"功能完全正常。

### 13.2 探路过程（5 轮）

#### 探路 1：写入探路（`写入探路.js`）

测试直接修改 `extensionSettings.mvu_settings` 对象 + `saveSettingsDebounced()`：

- `getMvuCfg()` 返回的是**真实对象引用**（两次读取 `===` 为 `true`）
- 修改 `cfg.更新方式` 后立即读取，值已变更 ✅
- `saveSettingsDebounced()` 调用后耗时 **1ms** 返回
- 保存后其他键（通知、额外模型解析配置等）**无丢失** ✅
- 恢复原值再保存，也正常 ✅

**结论**：直接改 cfg 对象 + save 机制本身没问题。

#### 探路 2：表单对比探路（`表单对比探路.js`）

逐一对比 22 个表单 DOM 元素当前值 vs `mvu_settings` 实际值：

- **24/24 全部匹配** ✅
- 兼容性 3 个复选框也全部匹配 ✅

**结论**：`syncMvuToForm()` 正常执行，表单回显无问题。

#### 探路 3：完整流程探路（`保存流程探路.js`）

模拟完整"应用"流程：改 cfg → save → 回读 → 对比键和值：

- 温度 1→0.7，保存后读取为 0.7 ✅
- 额外模型解析配置 17 键 → 保存后仍 17 键 ✅
- 顶层 6 键全部保留 ✅
- 丢失键数：0

**结论**：内存层面一切正常，问题不在 JS 逻辑层。

#### 探路 4：保存函数探路（`保存函数探路.js`）

搜索所有可用保存函数：

- iframe `SillyTavern` 只有 `saveSettingsDebounced`，**没有非 debounced 版本**
- 父页面无任何保存函数
- 源码：
  ```javascript
  (...args) => {
      clearTimeout(timer);
      timer = setTimeout(() => { func.apply(this, args); }, timeout);
      debounceMap.set(func, timer);
      debounceMap.set(fn, timer);
  }
  ```

**关键发现**：`saveSettingsDebounced` 只是一个 `setTimeout` 包装！调用后 1ms 返回，实际写盘在 `timeout` 毫秒后才执行。

#### 探路 5：保存机制深探（`保存函数探路2.js`）

- `writeExtensionField` 存在（3 参数），但源码显示它是写给**角色扩展数据**（`i.data.extensions`），非全局 `extensionSettings`
- `debounceMap` 是闭包变量无法外部访问，无法获取 `timeout` 具体值
- 全局没有其他保存函数

### 13.3 根因

```
"一键最优配置" 流程：
  applyOptimalMvuConfig() → 改 cfg → saveSettingsDebounced()
  → setTimeout 排期写盘 → showToast + 2秒后 reload
  → 2秒内写盘已完成 ✅ → 刷新后读盘拿到新值 ✅

"应用"按钮 流程（修复前）：
  writeMvuConfig() → 改 cfg → saveSettingsDebounced()
  → setTimeout 排期写盘 → 立刻 reload()
  → 写盘 setTimeout 被取消 ❌ → 刷新后读盘拿到旧值 ❌
```

**`saveSettingsDebounced` 用 `setTimeout` 延迟写盘，"应用"按钮立即 `reload()` 导致 setTimeout 被浏览器取消，改动从未落盘。刷新后自然回到磁盘上的旧值（可能是出厂默认值）。**

### 13.4 修复

"应用"按钮两处分支均改用和"一键最优配置"相同模式：

```javascript
writeMvuConfig();
try { await saveSettings(); } catch(e) {}
showToast('配置已应用，2秒后刷新页面...');
setTimeout(() => { window.parent.location.reload(); }, 2000);
```

### 13.5 教训

- `saveSettingsDebounced` 名中 "Debounced" 不只是一个约定名称——它真的是 `setTimeout` 延迟执行
- 所有需要保存后刷新页面的操作，必须给 debounce 留足时间（2 秒安全值）
- 直接改 cfg 对象（不经过表单 DOM）的路径（如"一键最优配置"）已经隐含了这个延迟，所以一直正常
